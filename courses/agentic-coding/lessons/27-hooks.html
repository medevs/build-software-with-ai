Building block #6, in depth

Hooks (Guardrails)

Every other building block asks the agent nicely to do the right thing. A hook doesn't ask â€” it's a plain command that fires automatically, whether the agent remembers to or not.

Ask nicely vs. enforce

A rules file (Lesson 22) tells the agent what it should do. That's powerful, but it's still a request â€” a very well-followed request, most of the time, but a probabilistic one. Under a long, complicated task, even a well-written instruction can get missed or deprioritized.

A hook is different in kind, not just in degree. It's a deterministic script wired to a specific moment in the agent's workflow â€” before a file gets edited, after the agent declares itself finished, and so on â€” and it runs automatically at that moment, every time, with no dependence on the agent choosing to comply. Nothing is being asked. Something is simply guaranteed to happen.

The distinction in one line Rules are how you ask an agent to behave. Hooks are how you guarantee it anyway, even if it forgets, gets distracted, or decides otherwise.

What a hook looks like in practice

The clearest example is validation. Instead of trusting the agent to remember "run the test suite before you say you're done," a hook can fire automatically the moment the agent finishes its work, run the tests itself, and force the agent to keep iterating if anything fails. The check isn't a step in the instructions the agent might skip under pressure â€” it's a script that runs no matter what.

A few other common shapes the same idea takes:

Blocking a risky edit

A hook can refuse an edit to a protected file â€” a migration, a production config â€” before it's ever written, regardless of what the agent intended.

Running the formatter

Instead of hoping the agent remembers to format code consistently, a hook runs the formatter automatically every time a file is saved.

"Don't stop until tests pass"

A hook can keep the agent in its loop, forcing another attempt, until a real command reports success â€” not until the agent merely claims success.

Logging what happened

A hook can record every command run during a session, so you have a plain factual trail, independent of any summary the agent gives you.

Why "deterministic" is the whole point

Everything else in your system leans on the agent being smart and cooperative â€” usually a safe bet, but never a certainty. Hooks exist for the handful of things you're not willing to leave to chance at all: the tests really did run, that file really was never touched, the formatter really did apply. A hook can't forget, get talked out of it, or decide the rule doesn't apply this time.

That also means hooks are cheap to reuse across projects. Like skills, a hook is behavior, not codebase-specific truth â€” "always run the formatter on save" is just as true on your next project as this one, so a well-written hook tends to travel with you.
