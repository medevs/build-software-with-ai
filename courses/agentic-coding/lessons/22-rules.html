Building block #1, in depth

Rules

Of the six building blocks you met earlier, rules are the one that pays off first and pays off biggest — because they're the one piece of your system that's entirely, specifically yours.

What a rules file actually is

A rules file is a single markdown document, usually sitting at the root of your project, that your agent reads automatically at the start of every single task. Nobody has to remember to paste it in. It's just there, every time, the way a new hire's onboarding doc should be but rarely is.

Inside it you write down the standing truth about your codebase: the stack you use, how the data is shaped, which boundaries not to cross, how you test, how you log, and where new code is supposed to go. Think of it as everything a good teammate already knows by feel after a few months on the project — written down once, so the agent doesn't have to guess or re-learn it every session.

Why this is the highest-leverage block Every other building block is somewhat generic — a skill or a hook can often be reused across projects almost as-is. A rules file cannot. It encodes your codebase's truth, so it's the one piece of your system nobody else could have written for you.

The one rule that matters more than any other

Rules must come from what your codebase actually does, never from what you wish it did. If you write "we always validate input at the API boundary" and half your endpoints skip that, you've just taught your agent something false — and it will confidently repeat the mistake back to you, dressed up as following the rules.

On an existing codebase, that means doing the legwork first: read through the code, or have the agent explore it, before you write a single rule down. On a brand-new project, this is the one exception — there's no existing truth yet, so it's fine to write down what you've decided the project should become.

Existing codebase

Derive rules from what the code really does. If you can't point to evidence of a rule in the codebase, it doesn't belong in the file yet.

Brand-new project

You're allowed to be aspirational here, because you're the one deciding the conventions as you go. Write down the decisions, not vague hopes.

Keep it lean, on purpose

Because a rules file loads on every task whether it's relevant or not, it competes for the same limited attention as everything else you hand the agent. A bloated rules file doesn't make the agent more careful — it makes the agent worse at everything, because the important lines are buried in restated framework advice and generic best practices it already knows.

This is the same idea covered back in Lesson 6 and the lesson on progressive disclosure: keep the always-loaded file short and high-level, and push anything long or situational into separate reference documents the agent only opens when a task actually calls for it. A rules file that tries to explain everything ends up explaining nothing well.

A habit worth building in Add a short "where new code goes" section — the handful of extension points where new work usually plugs in (a new endpoint, a new database column, a new integration). It turns "I have no idea where to put this" into a five-second lookup.

Write it once, recheck it occasionally

A rules file isn't a living diary you update daily. You write it once, up front, and it stays mostly stable. The one exception is a periodic drift check: every so often — a natural point is after a chunk of new work lands — you compare the file against the code again and prune anything that's gone stale. Code moves on; an unwatched rules file quietly stops being true, and a false rule is worse than no rule at all.

The one thing to remember

Your rules file is the one building block that's entirely, specifically yours — derive it from what the code truly does, and keep it lean. Get this one piece right and almost everything else your agent does gets easier.

Reflect & discuss Think about a mistake your agent keeps repeating. Is it because a rule is missing, or because the true rule is buried under too much noise in the file? Bring your answer to the course discussions — and if you want a starting point, there's a rules-file template in the course repo.
