Naming the thing you're actually going to build

The Context Layer

You already know a system beats a raw prompt. Now let's name the piece you build once and reuse forever: the context layer.

What the context layer actually is

Picture handing a new contractor the keys to your codebase with zero onboarding. No conventions, no workflow docs, no idea which systems talk to which. They'd guess, and some of those guesses would be wrong in expensive ways. A raw coding agent starts every session exactly like that contractor — capable, but blank.

The context layer is everything you build on top of the agent to fix that: the knowledge about your project, the workflows you want it to follow, and the connections to the other systems it needs to reach. Concretely, it's a set of files that live right alongside your code — conventions, reusable workflows, specialist helpers, connections to outside tools — checked into the same repository as everything else you ship.

Because it's checked in, it isn't personal to you. Open a new session tomorrow, or hand the project to a teammate next week, and the context layer is already there. Nobody re-explains the codebase from scratch, and nobody starts back at zero.

Three layers, one stack

It helps to see where the context layer sits relative to everything else in play when you're working with an agent. Think of it as three layers, stacked bottom to top:

1 · The agent you pick

The coding tool itself — Claude Code, Codex, Cursor, or whatever you reach for. This is the foundation. You don't build it, you choose it, and it's the first thing that turns a raw model into something that can read and write code.

2 · The context layer you build

Everything this lesson is about: the reusable knowledge, workflows, and connections you add on top of the agent. You build this once per project and keep improving it. This is the layer under your control.

3 · The plan for one task

Inside a single work session, you produce a focused, per-task plan — what to build, right now, for this one piece of work. It's temporary: useful for this loop, then done.

Why the order matters

Each layer leans on the one below it. A good per-task plan is easy to write when the context layer already carries the conventions and workflows. A reliable context layer is easy to build on top of a capable agent. Skip a layer and the ones above it get shakier.

Moldable, not a framework

It's tempting to picture the context layer as a rigid process you install once and follow forever — a set of steps you're locked into. Resist that. Some off-the-shelf approaches do work that way, forcing every project through the same fixed pipeline whether it fits or not. That's the opposite of what you want here.

The key move The context layer is yours to shape. Start small, notice where the agent stumbles, and adjust. It should feel more like a garden you tend than a machine you install.

That's also why it's worth starting light. A context layer that tries to cover every possible situation on day one becomes its own maintenance burden. A context layer that starts with a handful of true, useful conventions and grows only when reality demands it stays something you can actually keep up with.

Why this lives in your repo, not your head

The alternative to a context layer isn't "no structure" — it's structure that lives in one person's head, re-explained verbally every time someone new joins or a new session starts. That doesn't scale past one person, and it doesn't survive you being on vacation.

Kept in your headKept in the context layer
Re-explained every sessionLoaded automatically, every session
Lost when you're unavailableAvailable to any teammate, any time
Drifts silently as the project changesReviewed and versioned like the rest of your code

Put it in the repo, and the context layer becomes a shared asset that compounds — every session, every teammate, every future you gets the benefit of what you already figured out.

The one thing to remember

The context layer is what you build on top of your agent, kept in your repo so every session and every teammate reuses it — and it's yours to shape, not a framework to obey. The next lesson breaks it into the six concrete pieces you'll actually build.
