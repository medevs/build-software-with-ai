One ticket, start to finish

The Full Loop, End to End

Every lesson so far has covered one piece of the system. This lesson walks a single small ticket through all of them, in order, so you can see the whole shape at once.

One ticket, picked up from the slice

Say your slicing pass produced a small, well-bounded ticket: add a "resend confirmation email" button to the account settings page. It touches one screen, one existing email template, and one small backend action. One testable concern: click the button, get the email. That's exactly the kind of slice the core loop is built to consume — and this is what running it looks like from open to merged.

  1. Isolate the work. Before anything else, the work gets its own git feature branch, named after the ticket. Nothing for this change ever gets written straight onto the main branch — the branch is what keeps this one small change reviewable and revertable on its own.
  2. Research (optional). For a ticket this small and well-understood, research may add little — skip it. On a vaguer ticket, or in an unfamiliar part of the codebase, this step primes the agent on how the account settings page and the email system actually work today, before any plan gets written.
  3. Plan. A short back-and-forth turns the idea into a concrete plan: which file the button goes in, which backend function it calls, which existing email template it reuses, and — critically — exactly how you'll validate it (click it, confirm an email arrives, confirm a second click doesn't send two). This becomes a plan a stranger could implement without asking you anything.
  4. Save the plan, clear the context. The plan is committed somewhere durable, and the chat that produced it is closed. A fresh session opens with the plan as its only input — no leftover back-and-forth, no accumulated clutter, just the plan.
  5. Implement, in that fresh session. The agent writes the button, wires it to the backend action, and — as its own final step — runs the validation strategy the plan specified. This proves the feature functions: the button appears, the click fires, the email sends.
  6. Validate again, with fresh eyes. A second, separate pass — often a different session or even a different agent — reviews the actual change. This pass isn't asking "does it work?" anymore (the previous step already showed that). It's asking a different question: is this built well? Right naming, no leftover debug code, no obvious edge case missed, nothing that will bite the next person who touches this file.
  7. Open the change for review. The branch becomes a pull request (or equivalent change request), with a clear description of what changed and how it was validated.
  8. A human approves and merges. You read the small, focused diff — it's short enough to actually read, because it was sliced that way from the start — and merge it into the main branch. The ticket is done; you move to the next one.
Why the split matters Steps 6 and 7 look similar but ask different questions on purpose. Implementing proves the app works. Reviewing proves the code is good. Folding them into one pass means one of the two questions quietly stops getting asked.

What made delegating all of that safe

Notice what the agent never did alone: it never decided what "done" means, and it never got the final word on whether the result was good enough to ship. The agent wrote every line of code, but that delegation was sandwiched — a human-owned plan on one side, a human-gated validation on the other. That sandwich is the entire reason it's safe to hand over the typing.

It doesn't have to be perfect The loop doesn't need a flawless run to be worth using. A plan that's slightly off, or a first implementation that needs a second pass, still beats no structure at all — each run's rough edges are exactly the kind of signal that improves the system for next time.

The one thing to remember

The full loop is one ticket, walked end to end: research, plan, clear context, implement, validate twice, review, merge. It's the same core loop from Module 3, made concrete — the exact sequence every shipped change in this system is made of.
