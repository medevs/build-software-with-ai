The highest-leverage move you'll make, and it isn't writing code

Evolving the System

When the same mistake keeps happening, the fix isn't a better fix. It's a better context layer.

The mistake that keeps coming back

You've probably felt this already: the agent makes a mistake, you correct it, and next session it makes the same mistake again. Maybe it keeps using the wrong naming pattern, or skips a validation step you always want run, or reaches for a library your team stopped using. Each time, you patch it in the moment — and each time, the underlying cause is still sitting there, waiting to bite the next session too.

That's the signal to pay attention to: not "the agent got this wrong," but "the agent got this wrong again." A single mistake is just a mistake. A repeated mistake is telling you something about your system.

Fix the system, not the bug

The key move When a mistake recurs, stop building for a moment and have a conversation about what to change in your context layer — usually your rules — so that whole class of failure gets less likely, not just this one instance of it.

This is a different move than fixing the bug in front of you. Fixing the bug gets today's code correct. Fixing the system means the next ten times that situation comes up, the agent is less likely to trip on it at all — because you wrote the lesson down somewhere it'll actually see it again.

In practice this usually means sharpening a rule: adding the convention you keep having to restate, or tightening a constraint that was too vague to actually prevent the mistake. Sometimes it means adjusting a skill's steps, or adding a new guardrail. The exact fix varies, but the target is always the same — the context layer, not the one file you're editing right now.

Why this compounds

The context layer is the one thing about your setup that you fully own and carry forward. Every time you fold a lesson back into it, every future work session starts from a slightly better baseline — not just for you, but for anyone else working in the same project.

Hand-fix only

You fix today's instance. Tomorrow, the same class of mistake is exactly as likely as it was yesterday. You're spending effort but the baseline never moves.

Fix the system

You fix today's instance and update the context layer. Tomorrow, that class of mistake is rarer. The baseline moves up, and it stays up.

Nothing about your starter context layer is meant to be final. Rules, skills, and the rest are a beginning, not a finished product — the expectation is that you'll keep shaping them as you learn where your project's real trouble spots are.

Don't overcorrect This isn't a call to stop and rewrite your rules after every tiny slip. Most of the time you keep building — the loop doesn't need to run perfectly to keep moving. Reach for a system fix specifically when something goes wrong more than once, or when it's clearly a big deal even the first time.

Two loops, one habit

This is the same "outer loop" idea from earlier in the course, now made concrete: most of your time is spent in the inner loop, actually building. But every so often, a recurring problem is your cue to step out of that loop briefly, patch the context layer, and step back in with a slightly better system than you had before.

The one thing to remember

When a mistake recurs, fix the context layer, not just the code. That's the move that makes every future session start from a better place than the last one.
