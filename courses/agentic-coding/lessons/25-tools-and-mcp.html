Building block #4, in depth

Tools & Connections (MCP)

Your agent can only reason about what it can reach. Tools and connections are how it stops guessing about the outside world and starts reading and acting on it directly.

Giving the agent arms, not just a brain

By default, an agent only knows what's in the codebase in front of it and whatever you type into the conversation. Ask it what a ticket says, what real users are actually doing, or what's currently sitting in a database table, and without a connection to those systems, it can only guess — or ask you to paste the answer in yourself.

A tool connection closes that gap. It wires the agent directly into an external system — a ticket tracker, a database, an analytics platform, the open web — so it can read (and sometimes write) real data instead of working from secondhand descriptions. Think of it as giving the agent arms to reach outside its own sandbox.

MCP: one standard, many connections

Wiring up a new external system used to mean custom, one-off integration work for every single tool. The Model Context Protocol (MCP) fixes that by giving agents and tools a single shared way to talk to each other — a common connector, the same idea as a universal port that many different devices can plug into. Once a system exposes an MCP server, any MCP-capable agent can connect to it the same way.

You can read the open standard itself at modelcontextprotocol.io. You don't need to memorize the protocol's internals — the useful takeaway is simpler: MCP is why "connect my agent to my ticket tracker" is now a small, repeatable setup step instead of a custom project.

Setting one up is usually simple Most platforms that offer an MCP server publish straightforward setup docs. You typically don't need to hand-write the integration — you can often just point your agent at the docs and have it configure the connection and handle authentication for you.

What this looks like in practice

A few concrete, beginner-friendly examples of what a connection actually buys you:

Reading a ticket

Instead of copy-pasting a ticket's description and comments into the conversation, the agent fetches it directly from your tracker by its ID — always current, never stale.

Querying usage data

Ask the agent to check what a product-analytics platform actually shows about how a feature is used, instead of guessing from a hunch — real numbers instead of assumptions.

Checking a database

The agent can query a database directly to see the actual shape or contents of your data, rather than inferring it from code that might be out of date.

Reading documentation

Pull a requirements doc, a design page, or a spec straight from wherever your team keeps it, instead of relying on someone's memory of what it said.

Connections are optional, and added when they earn their place

You don't need every connection on day one. Add a tool connection when a real, recurring need shows up — you keep manually pasting ticket text in, or you keep wishing the agent could just check the data itself. That mirrors the broader lesson from Lesson 6: rules and skills are where you start, and the remaining blocks, including connections, tend to get added once you actually feel the friction they solve.

What problem does MCP actually solve?

Without any tool connections, how does an agent answer a question about a live ticket or a database?

The one thing to remember

Tool connections give your agent arms to reach real external systems, and MCP is the common standard that makes wiring one up simple. Add a connection when you notice yourself feeding it the same outside information by hand, over and over.
