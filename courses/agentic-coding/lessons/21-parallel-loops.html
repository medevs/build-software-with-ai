One loop at a time was the training wheels

Running Loops in Parallel

Once you trust the loop on one ticket at a time, the natural next question is: why not run several at once? The answer is a git primitive called a worktree.

What you're actually scaling

Your slicing pass already produced a dependency map: some tickets must land before others, but plenty of them don't depend on each other at all. Those independent tickets don't have to be built one after another — they can be built by separate agent sessions running at the same time.

Be precise about what that buys you. It's throughput, not cost: the same total amount of work gets done in less wall-clock time. It isn't free extra output, and it isn't several agents fighting over the same task — it's independent tickets, independently built.

The problem parallel sessions create

Two agent sessions working in the very same folder is asking for trouble. One session edits a file; the other, still holding an older version of that file in its head, edits it too — and now one of them is working from information that's already gone stale, or the two edits collide outright. This is the same context-hygiene problem from Clean Context, just crossing from a single session into several sessions sharing one workspace.

The failure mode Without isolation, agents step on each other's edits, override each other's changes, or act on files that have silently mutated underneath them.

The fix: a git worktree

A git worktree is a second, full working copy of your repository, checked out on its own branch, that still shares the single underlying git history. Run git worktree add <branch> and you get a sibling folder holding the entire project — your code, your configuration, everything a coding agent needs — on a brand-new branch created for it automatically. (git-scm docs)

That gives you two levels of isolation, and both matter for a different reason:

A separate file copy

Stops the file-collision problem outright. Each agent reads and writes its own copy of the code — nothing mutates underneath it, and nothing it edits can collide with what another session is doing.

A separate branch

Keeps each stream of work as its own clean slice of git history, so one ticket's change can be reviewed, revised, or abandoned without touching any other ticket's progress.

In practice, you open an agent inside each worktree folder and prompt it on its own ticket. While one agent grinds through its task — often somewhere around twenty minutes for a well-sized ticket — you switch over and prompt the next one. Nothing about the loop itself changes: each worktree is simply running its own full loop, exactly as covered in the last lesson.

When it helps, and when it's just overhead

Parallel loops shine when you have several independent, well-sliced tickets ready to go and the patience to keep more than one thread of context organized in your own head. They add nothing — and can actively hurt — when tickets aren't actually independent, when a ticket isn't sliced cleanly enough to hand off with a solid plan, or when you're still building trust in the loop itself. Running five shaky loops at once just gets you five shaky results, faster.

The real bottleneck It isn't compute or tooling — it's how many parallel threads a human can actually hold in their head without losing track of any of them. Start with two. Grow the muscle toward three, four, or five as it gets comfortable, rather than jumping straight to the max.

You rarely need to type the raw git commands yourself. Standing up several worktrees involves setup a bare worktree skips — copying local configuration, installing dependencies, confirming everything actually runs — and merging finished work back cleanly, including resolving conflicts when two branches touched the same file. Both of those chores are exactly the kind of repeatable workflow this course later shows you how to package once and reuse.

The one thing to remember

A worktree gives each parallel loop its own isolated copy of the repo and its own branch. Parallelism buys you throughput on independent, well-sliced tickets — and the bottleneck it runs into first is never the tooling, it's how many threads you can hold in your own head.
