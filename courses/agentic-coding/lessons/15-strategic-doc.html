Write down the what and why, before any code

The Strategic Doc

One document sits above everything else you'll plan: the strategic doc. It captures intent — the problem, the users, the bet — and deliberately says nothing about how to build it.

The document at the top of the stack

In the last lesson you met the three altitudes: intent, architecture, and task. The strategic doc lives at the very top — intent altitude. If you're starting something brand new, it's a PRD (product requirements document): what should this product be? If you're adding a feature to something that already exists, it's an epic: what's the next meaningful slice of work? Same shape, same job, just a different starting point.

The key move A strategic doc answers the what and the why. It never answers the how — that's a different document, one altitude down, written next.

What it holds

Whether it's called a PRD or an epic, a good strategic doc has the same core sections:

Problem & evidence

What's actually broken or missing today, and what makes you believe that, not just guess it.

Users

Who this is for, specifically enough that "everyone" isn't a valid answer.

Success

How you'll know it worked — ideally a falsifiable hypothesis, which the next lesson covers in depth.

Scope & non-goals

What's explicitly in for this pass, and just as important, what you're deliberately leaving out.

Notice what's missing from that list: the stack, the data model, the components, the file layout. None of that belongs here. Keeping the how out is not laziness — it's what keeps the doc reviewable by anyone who cares about the product, without wading through engineering detail that isn't their job to weigh in on.

Why it's worth the extra step

It's tempting to skip straight from an idea to code, especially with an agent that will happily start typing the moment you describe something. But every plan that follows — the architecture, the tickets, the actual implementation — inherits its assumptions from this doc. One wrong guess about who the user is, or what "done" means, doesn't just cost you one bad line of code. It cascades: it becomes a wrong architectural boundary, which becomes a dozen wrong tickets, which becomes hundreds of lines built confidently in the wrong direction.

The strategic doc exists so that every later loop of work has something solid to "grab onto" — shared context that doesn't have to be re-explained or re-guessed each time you sit down with an agent. Write it once, well, and reference it from everywhere downstream.

A repeatable shape beats a clever one

Resist the urge to freestyle a new structure every time. The value of a strategic doc comes largely from consistency: the same sections, in the same order, every single time you write one. That predictability is what makes the doc easy to review quickly, easy for an agent to parse when it's grounding later work in it, and easy for you to trust that nothing important got skipped.

The one thing to remember

The strategic doc is the what and the why, written down once, in a repeatable shape, before any code exists. Everything you plan afterward leans on it.

Reflect & discuss Think of a feature you've seen built (or built yourself) without a clear problem statement up front. What went wrong, and would a short strategic doc have caught it? Bring your answer to the course discussions; comparing notes with other learners is where a lot of the real learning happens.
