Turn a conversation into an artifact

The Plan That Ships in One Pass

A great plan isn't a summary of a chat — it's a self-contained document detailed enough that an agent who never saw the conversation could build the task correctly from it alone.

Two very different kinds of planning

The Plan step of the core loop actually happens in two distinct moves, and mixing them up is where a lot of people go wrong.

The first move is loose and conversational — think of it as a brain-dump. You describe the task in your own words, half-formed thoughts and all, and let the agent research the relevant area, propose a couple of approaches, and — importantly — ask you clarifying questions back. This is deliberate, messy exploration that you stay in control of the whole time, not "type a vague request and hope."

The second move is where that loose conversation gets converted into a structured, written plan — a real document, not a chat transcript. That document is what actually goes on to be implemented.

Not the same thing A rambling chat where you eventually agree on an approach is not a plan. A plan is the document you write afterward that captures what you agreed on, in a form someone else could pick up and act on.

The bar: the "no prior knowledge" test

Here's the single best way to judge whether a plan is actually good enough: imagine handing it to a completely fresh agent — one that never saw your planning conversation, has no memory of your brain-dump, and knows nothing except what's written on the page. Could that agent implement the task correctly from the plan alone?

If the answer is "well, it would need to ask a few things first," the plan isn't done yet. If the answer is "yes, everything it needs is right there," you're ready to implement.

Why this bar matters so much Passing this test is exactly what lets you clear the conversation and start implementation in a brand-new session — a move you'll learn about next lesson. A weak plan makes that reset unsafe.

What a one-pass-ready plan actually contains

There's no single required format, but a plan that passes the "no prior knowledge" test tends to cover the same ground every time:

  • Goal and what success looks like. What this task is for, in plain terms, and what "done" means.
  • Real file paths. Not "update the user settings area" but the actual files and locations involved — specific enough that there's no guessing.
  • Task-by-task steps. The concrete list of changes, in an order that makes sense to build in.
  • Explicit out-of-scope. What this task deliberately does not touch, so the agent doesn't wander into adjacent work.
  • A validation strategy. Exactly how the agent will prove, on its own, that the work is correct.

    • Of all of these, the validation strategy is the most important. It's the section worth iterating on the most, because it's what turns "I hope this works" into "here's how we'll know it works" — the difference the next two lessons build directly on.

    Why the questions matter During the brain-dump conversation, let the agent ask you multiple-choice-style clarifying questions rather than open-ended ones. It's much faster to pick "A or B" than to write a paragraph, and it forces the agent to surface its assumptions instead of quietly guessing.

    The one thing to remember

    A plan only counts once it passes the "no prior knowledge" test. Real file paths, explicit out-of-scope boundaries, and a concrete validation strategy are what get it there — and what make the next step, working from a completely fresh session, safe to do.
