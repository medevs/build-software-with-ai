Decide the shape before you slice the work

Architecture Decisions

The strategic doc says what and why. Before any ticket gets written, something else has to say how — the high-level shape of the solution. That's the architecture decisions doc, and it lives at the middle altitude.

The twin of the strategic doc

Recall the three altitudes: intent, architecture, task. The strategic doc from a couple lessons back covers intent. Architecture decisions cover the next one down — the stack you'll use, the key components, how data flows between them, and where this work plugs into anything that already exists. Think of the two documents as twins: linked, but deliberately kept separate, so a reader focused on the product doesn't have to wade through engineering detail, and a reader focused on engineering doesn't have to re-litigate the product bet.

The key move Architecture decisions answer "what's the shape of the solution," not "what exact code do we write." That's still one altitude further down, at task level, per ticket.

Decisions, not tasks

It's tempting, once you're deciding the how, to keep going straight down into pseudocode, schema diffs, and file-by-file plans. Resist that. An architecture doc deliberately stops one level above implementation. It records the choice — "we'll add a new adapter here, following the existing pattern" — and leaves the exact lines of code to the per-ticket planning that happens later, one small slice at a time. Stopping here keeps the doc a decision record, not a task list, and keeps it useful even as the fine-grained plan around it changes.

Ground it in the real codebase

An architecture decision made in the abstract, without looking at the actual code, is a guess wearing a suit. The useful version of this doc comes from holding two things side by side: the actual codebase — its existing structure, its established patterns, the places new code is meant to plug in — and the strategic doc's intent. Decisions that ignore the first tend to look great on a whiteboard and fall apart on contact with a real file tree.

In practice, that means the decisions here should point at real seams: the boundaries and extension points your project already has, documented in your project's own rules. New work should plug into those known seams rather than inventing new ones for no reason. When it does need to invent something new, that's worth calling out explicitly as a decision, not sliding past quietly.

What the doc contains

Options considered

The two or three real approaches you weighed, briefly, so the final call has visible reasoning behind it instead of appearing out of nowhere.

The recommended approach

The stack, the key components, and how data moves between them — the actual shape you're committing to.

Integration points

Where this plugs into what already exists, grounded in the codebase's real seams.

Door decisions

Which calls are cheap to reverse and which aren't — the subject of the next lesson.

Use the template A ready-made architecture decisions template is available in the course resources: architecture template. Start there instead of a blank page.

Why decide this before slicing tickets

Once architecture decisions are settled, every ticket that comes out of slicing the work inherits the same shape — the same stack, the same boundaries, the same conventions — instead of each ticket quietly reinventing its own. That consistency is what lets several small, independent loops of work stay compatible with each other later, even when they're implemented separately, sometimes weeks apart. Skip this step, and you get a codebase that works, piece by piece, but doesn't agree with itself.

The one thing to remember

Decide the high-level shape — stack, components, data flow, integration points — before you slice any tickets, and ground every decision in the real codebase. That's what keeps the individual loops of work consistent with each other.

Reflect & discuss Think of a project where two parts of the codebase clearly disagreed with each other — different patterns for the same kind of problem. Would an architecture decisions doc, made once up front, have prevented that? Bring your answer to the course discussions; comparing notes with other learners is where a lot of the real learning happens.
