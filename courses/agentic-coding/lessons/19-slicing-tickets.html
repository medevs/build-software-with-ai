From a big idea to small, buildable pieces

Slicing the Work

You've decided what to build and how. Now comes the step people skip: cutting that decision into pieces small enough for one loop to build well.

Why you can't just hand over the whole thing

Imagine you've just finished deciding the shape of a new feature — the "what" and the "how" are both settled. The tempting move is to paste the whole thing into a chat and ask the agent to build all of it in one go. This is close to the worst way to use a coding agent.

An agent working on one enormous task has to hold too much in its head at once. The more it's juggling, the more it starts guessing, skipping details, or quietly drifting from what you actually meant — the same "dumb zone" that shows up whenever a context window gets overloaded. And a giant task removes the one thing that makes delegation safe: a point where you, the human, look at a small result and say yes or no before more work piles on top of it.

The rule A slice is bounded not by how many lines of code it takes, but by one testable concern — one result a human can actually check once the agent is done.

Capacity is not permission

Modern coding agents can hold a huge amount of context. That's exactly the trap: just because a whole project would technically fit in the context window doesn't mean it should be run as one piece of work. Fitting is not the same as being wise to run that way.

A ticket sized for an agent can be far bigger than one sized for a human — think roughly eight to ten subtasks, five hundred to fifteen hundred lines of change — because the agent handles more mechanical work per pass than a person typing by hand. But it must still resolve to one thing you can validate. That's the discipline: plan the whole project up front, at a wide angle, then cut it thin before any code gets written.

Three ways to cut a slice

There's no single correct way to split work — but three approaches cover almost every case.

ApproachHow it worksWhen to reach for it
By dependencyOne piece has to exist before the next can be built on top of it.Sequencing work, and pulling riskier pieces earlier so you learn fast if something won't work.
By concernGroup by the part of the codebase touched — the database schema, the API, the front-end.The simplest and most common split; a good default when you're not sure.
Thin end-to-end sliceOne narrow feature that cuts through every layer at once — database to UI.Rarer, more technical; useful when you need to prove a whole path works before widening it.

Whichever way you cut, the output is the same shape: a set of tickets, each one testable concern, with the dependencies between them mapped out. Some of those tickets can run one after another; others, once you trust the process, can run at the same time. Where those tickets live doesn't matter — a tracker like Jira or Linear, GitHub Issues, or just a folder of markdown files all work.

Signs a slice is still too big

You can't state the check

If you can't say, in one sentence, what "done" looks like for this ticket, it's still bundling more than one concern. Split it until you can.

It touches unrelated layers for no reason

A ticket that edits the database schema, rewrites an unrelated UI screen, and adds a new integration is really three tickets wearing one coat.

"And" keeps showing up in the title

"Add the endpoint and update the settings page and fix the old bug" — each "and" is usually a seam where the ticket should split.

You can't imagine reviewing it in one sitting

If the resulting change would be too large to review with fresh eyes in one pass, it was too large to build in one pass, too.

What slicing is not Don't deep-plan each ticket at slicing time. Dependencies aren't fully knowable until upstream tickets actually exist, and priorities shift as you go — so the detailed, one-pass-ready plan for a ticket gets built inside the loop that picks it up, right before implementation, as covered in The Plan That Ships in One Pass. Slicing just decides the boundaries and the order.

The one thing to remember

One testable concern per ticket. Slicing is what turns a big decision into clean raw material for the loop — plan wide, then implement narrow, one small, checkable piece at a time.

Reflect & discuss Think of a recent task you handed to an agent that came out messy. Was it really one testable concern, or was it secretly two or three? Bring your answer to the course discussions — comparing notes with other learners is where a lot of the real learning happens.
