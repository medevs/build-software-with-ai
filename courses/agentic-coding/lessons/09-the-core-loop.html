The one rhythm you'll run hundreds of times

The Core Loop

Four steps, one small task at a time: Research, Plan, Implement, Validate. Everything else in this course is detail around this one rhythm.

Meet the loop

Every reliable build with a coding agent, no matter how small or how big the codebase, comes down to the same four-step rhythm repeated over and over:

  • Research. Build the understanding a task needs before you plan it — the relevant code, the domain, the constraints.
  • Plan. Turn a loose conversation about the task into a written, structured plan the agent can follow.
  • Implement. Hand that plan to a fresh session and let the agent write the code.
  • Validate. Prove the work is correct — and separately, that it's well built — before anything ships.

    • We'll call this the core loop. You'll see it abbreviated all over this course, and you'll run it on nearly every single task you hand to an agent from here on.

    Why it's called that It's the most stable idea in this entire course — the piece that changes least as tools and models evolve. Learn this rhythm well and you can adapt to any new agent, any new project, any new model.

    Why one small task, never a whole project

    The single most important rule attached to the core loop is scale: you run it on one granular task at a time — a single well-defined piece of work, not a sprawling feature, not a whole project, not a stack of unrelated changes bundled together.

    Throwing an entire project at an agent in one go is one of the most reliable ways to get the worst possible results. The agent loses track of what actually matters, makes conflicting decisions across the file tree, and you end up with something that "sort of" works but that nobody — including the agent — can explain. Break the work down first (we'll cover how, later in the course), then run the loop on each piece.

    Common mistake "Build me the whole feature" is a passenger move. "Here's one task, here's the plan, go" is a driver move. Small scope is what makes every other part of the loop work.

    A quick preview of each step

    Each of the four steps gets its own lesson right after this one, but here's the shape of what's coming:

    StepQuestion it answersWhen it's essential
    ResearchWhat do I need to understand first?New or vague work; skippable when the task is already well understood
    PlanWhat exactly will be built, and how will we know it works?Always — this is the step that makes everything downstream safe
    ImplementWho writes the code, and from what?Always — runs from the plan, not from memory of the conversation
    ValidateDoes it work, and is it built well?Always — the last gate before anything merges
    A useful analogy Think of a contractor building a room addition. They don't start swinging hammers the moment you mention the idea. They walk the site (research), draw up a plan you both sign off on, build to that plan, then have an inspector check the work before you move furniture in. Skip any one of those and you're gambling with the whole house.

    Why this loop, and not just "chat until it works"

    Chatting back and forth with an agent until something looks right feels productive, but it produces work that's hard to reproduce, hard to hand off, and hard to trust. The core loop turns that same effort into something repeatable: a written plan you can reread, a fresh implementation session that isn't biased by a long meandering conversation, and an explicit check for whether the result is actually correct.

    That repeatability is what lets you scale from one task to many — and eventually, run several of these loops side by side without losing your grip on any of them.

    The one thing to remember

    Research, Plan, Implement, Validate — run on one small task at a time. Master this rhythm and every tool change, every model upgrade, every new project becomes something you can adapt to instead of something that throws you off.
