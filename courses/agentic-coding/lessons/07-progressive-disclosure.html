Why more isn't always better context

Progressive Disclosure

If you were tempted to cram everything the agent might ever need into its always-loaded context, this lesson is the reason not to.

Context is scarce, not free

It's tempting to think "more context is always better" — if a little background helps the agent, surely a lot helps more. It doesn't work that way. An agent's attention is a limited resource, much like a person's. Load it up with pages of detail before it has even seen the task, and the important parts get buried in the unimportant ones. Past a certain point, more upfront context makes the agent's judgment noticeably worse, not better.

The trap Writing a giant, all-inclusive rules file feels thorough. In practice it competes with itself — the agent has to wade through everything you ever thought might matter just to find the one convention relevant to the task in front of it.

The fix: two tiers, not one pile

The fix isn't to write less down. It's to organize what you write into two tiers:

Always-loaded

A small set that ships into every single conversation, whatever the task. Kept short and high-level on purpose — just enough for the agent to orient and to know what exists.

On-demand

Everything else: the heavier detail, the edge cases, the task-specific instructions. It only gets pulled in when a task actually calls for it.

Think of it like a table of contents versus the chapters behind it. You don't hand someone the entire book open to every page at once — you hand them the table of contents, and they turn to the one chapter they need. The rest of the book is still there, fully available, just not competing for attention until it's relevant.

Where it shows up in your context layer

Two of the six building blocks from the last lesson are built around exactly this idea:

  • Skills only show the agent a name and a one-line description up front — that's the always-loaded part, and it's just enough for the agent to recognize "this is relevant" and decide to use it. The full step-by-step instructions only get read once the skill is actually invoked.
  • Rules work the same way. The always-loaded rules file stays short and high-level, while detail specific to one kind of task — a particular workflow, a tricky subsystem — lives in a separate reference document that the rules file points to but doesn't paste in. The agent only opens that reference when the task in front of it matches.
Why this compounds Doing the heavy explaining once, up front, in a well-organized reference means the agent never has to re-explore or re-learn it from scratch in a later session. You pay the cost of writing it once; every future task gets to borrow it for free, without paying the attention cost of loading it every single time.

A habit that makes this easier

Progressive disclosure gets much easier to maintain if you keep a consistent structure across your documents — the same section headings, in the same order, every time. Predictable structure is what lets you (and the agent) jump straight to the slice that's needed instead of reading a whole document to find it. It's a small habit, but it's what makes "load only what's relevant" actually practical instead of just a nice idea.

Check yourself

Why doesn't stuffing every possible detail into the always-loaded rules file make the agent smarter?

What does a skill show the agent before it's ever invoked?

The one thing to remember

Keep the always-loaded context small, and push everything else into on-demand detail the agent pulls in only when a task needs it. That's how skills and rules stay useful instead of becoming their own burden.
