Building block #5, in depth

Code Navigation & Search

In a small project, blind text search is good enough. In a large one, finding the wrong code costs you time, tokens, and trust — and better navigation is the fix.

Two very different ways to "find" code

Imagine you need to see everywhere a particular function is used before you change its signature. There are two very different ways an agent can go about that:

Blind text search

Search for the function's name as plain text across every file. It's fast to set up, but it has no idea what it's actually looking at — it will match a comment that mentions the name, a different function that happens to share it, and a string that isn't code at all.

Symbol-level navigation

Jump straight to where a symbol is defined, and list every real reference to it — the same way "go to definition" and "find references" work in a modern code editor. It understands the code's actual structure, not just its text.

The second approach is what this building block is about: giving the agent the same precise, structural way of moving through code that a good editor gives a human developer, instead of leaving it to guess from plain text matches.

Why blind search quietly breaks down

On a small project, the difference barely matters — there's so little code that even a clumsy search usually turns up the right answer eventually. The trouble shows up as a codebase grows: common names get reused in dozens of unrelated places, similar-looking functions live in different modules, and a plain-text match starts returning far more noise than signal.

When that happens, two costs stack up. The agent burns time and context sifting through irrelevant matches before finding the real one — context that could have gone toward actually solving your problem. And worse, it sometimes edits the wrong match entirely, confidently changing a function that merely looked right instead of the one you meant.

Where this bites hardest The bigger and older a codebase gets, the more it depends on precise navigation instead of text matching. A pattern that works fine on a small app can quietly produce wrong edits on a large one.

Why this matters for reliability and cost

Precise navigation pays off in two ways that are easy to underrate until you feel them:

BenefitWhy it matters
ReliabilityThe agent finds the actual definition and actual call sites, not a lookalike — so the change it makes is the change you meant.
Token costLess time spent sifting irrelevant search results means less wasted context — more of the agent's attention goes toward your actual problem.

Neither of these is dramatic on a small project. Both become the difference between a smooth session and a frustrating one once a codebase is large enough that plain-text search starts turning up more noise than signal.

You don't need to build this yourself

Better code search usually isn't something you write from scratch — it's a capability you enable, often through the same tool connections covered in Lesson 25. The practical move is simply knowing this building block exists and reaching for it once your project is big enough that you start noticing the agent finding the wrong thing, or spending visibly long stretches searching before it acts.

The one thing to remember

In a large codebase, precise symbol-level navigation beats blind text search — it finds the right code instead of a lookalike, saving both reliability and tokens. Reach for it once your project is big enough that plain search starts missing.
