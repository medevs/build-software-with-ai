Move the loop off your screen

Scaling & Automation

Once you trust a loop to run reliably, the next step isn't running it faster by hand — it's running it without your hands at all.

From "I run the loop" to "the loop runs"

Earlier in the course you learned to run one core loop at a time, then to run several loops in parallel using isolated copies of your codebase. That's already a big jump in throughput — but it still requires you, at a keyboard, starting and watching each one. The next stage of scaling asks a different question: which of these loops could run without you present at all?

A surprising number can. If a piece of work is well-scoped, has clear rules to follow, and has a way to check itself when it's done, it doesn't strictly need a human staring at the terminal while it runs. It needs a trigger, a place to run, and a way to report back.

Three ways to move loops off your screen

Background agents

Kick off a loop and let it keep working while you do something else — write another ticket, review a different piece of code, or step away entirely. You come back to finished work instead of babysitting a progress bar.

Remote or cloud agents

Run the loop on a machine that isn't your laptop. This frees you from keeping your own machine on and open, and lets several loops run in parallel without competing for your local resources.

Automated loops in CI

Trigger a full Research → Plan → Implement → Validate pass automatically as part of your normal development pipeline — for example, on every pull request, using a continuous-integration platform such as GitHub Actions.

Notice what doesn't change: it's still the same core loop, following the same rules your context layer defines. What changes is where and when it runs, and whether a person has to be watching.

A concrete example: the loop on every pull request

Imagine a rule that says every pull request gets an automatic pass: the agent reads the diff, checks it against your project's rules, runs your tests, and leaves a review comment before a human ever opens the PR. That's the same Validate step you've been running by hand, just wired into your CI pipeline so it fires on its own trigger instead of a command you type.

Why this matters This is the payoff of everything earlier in the course. A loop that only works when you personally drive it is a personal habit. A loop reliable enough to run unattended, on a schedule or a trigger, is a system — one that scales beyond one person at one keyboard.
Start small You don't need a fully automated pipeline on day one. Automate the smallest, safest loop first — say, running your test suite and a rules check on every pull request — and only widen the scope once you trust what comes back.
A caution Automation multiplies whatever you feed it, same as everything else in this course. A loop with weak rules or no real validation step will now produce weak results automatically and at scale. Get the loop right by hand first; automate it second.

The one thing to remember

Scaling isn't running the loop faster — it's running the loop without you. A reliable loop can move from your keyboard to the background, to a remote machine, to an automatic trigger in your pipeline, and still do the same trustworthy work.
