Breaking the layer into pieces

The last lesson named the context layer as a whole. In practice, that layer is made of six separate kinds of building blocks. Each one answers a different question about how the agent should behave, and each lives as its own kind of file or connection in your project. You'll get a full lesson on each of these later â€” for now, the goal is just to recognize the shape of all six and know which ones to reach for first.

Start with rules and skills

Six pieces is a lot to build at once, and you don't have to. Of the six, two carry almost all of the early value:

Rules

These are the biggest lever because they're the most specific to your codebase â€” its conventions, its constraints, the things every teammate already knows by feel but nobody wrote down. Get these right and the agent stops making the same avoidable mistakes.

Skills

Almost any workflow you find yourself repeating â€” a planning routine, a review pass, a setup checklist â€” can become a skill. This is usually the highest-leverage thing you can invest in, because one skill saves you from re-explaining the same process forever.

The remaining four â€” subagents, external connections, code search, and hooks â€” matter too, and you'll build each of them out later. But they tend to get added as you actually feel the need for them, often while you're deep in a work session and notice a repeated pain point. Rules and skills are the two you reach for first because they pay off immediately, on almost every project.

One more thing they share

None of these six blocks are tied to a specific coding tool. Whether you're working in one agent or another, the same six ideas apply â€” rules are still rules, a skill is still a reusable workflow, a hook is still a guaranteed script. That's what makes the context layer portable: you're not learning a tool's quirks, you're learning six ideas that outlive any single tool.
