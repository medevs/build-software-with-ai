Two questions, never one

Validate & Gates

"Does it work?" and "Is it built well?" are two completely different questions. Keep them separate, and never let the human step be optional.

Don't get the agent's first draft

The final step of the core loop is Validate, and the first thing to understand about it is what it isn't: it isn't "read whatever the agent produces on its first try and decide if you like it." A good implementation step keeps working after the first draft — running checks, catching its own mistakes, and fixing them — before a human ever looks at the result.

That self-correction is only possible because the plan from two lessons ago already spelled out a validation strategy. The agent isn't guessing what "correct" means; it's checking its work against a bar you set in advance.

Two separate questions

Validation actually splits into two distinct questions that should never get merged into one pass:

Does it work?

This is validation — a functional question. Does the code do what it's supposed to do? This is what the agent checks and self-corrects on, before anyone looks at the result.

Is it built well?

This is review — a quality question. Is the code well structured, maintainable, and free of shortcuts that will bite later? This needs a genuinely fresh perspective, not the same conversation that wrote the code.

Why keep them apart Mixing "does it work" with "is it well built" tends to produce shallow versions of both — a quick functional glance that also tries to judge architecture, and neither gets done properly. Treat them as two separate passes with two separate goals.

The validation pyramid

"Does it work?" is best answered cheapest-first. Picture it as a pyramid: broad, fast checks at the base, narrower and more expensive checks as you go up.

  • Lint and type checks. Fast, mechanical, catches obvious mistakes immediately.
  • Unit tests. Confirms individual pieces of logic behave as expected.
  • Integration checks. Confirms those pieces work together correctly.
  • End-to-end. The agent actually drives the real, running application the way a user would, confirming the whole path works.

    • The agent runs this whole pyramid itself and fixes what it finds before handing anything back. Cheap and broad checks first means problems get caught in seconds, not after a slow, expensive end-to-end run.

    Why review needs a different session

    Once something passes validation, it still needs the second question answered: is it actually built well? That judgment shouldn't come from the same conversation that wrote the code — for the same reason a student shouldn't grade their own exam. A session that's been immersed in "how I built this" naturally defends its own choices.

    So review runs separately, with fresh eyes that only see the finished result — the same instinct behind starting implementation in a clean session, applied one step later in the process.

    Recall check This should sound familiar: it's the exact same idea as the "no prior knowledge" bar from the plan — a fresh perspective, with no bias from what came before, gives you the most honest judgment.

    The human is the final gate

    None of this — self-correction, the pyramid, fresh-eyes review — replaces a human making the final call. After the agent's checks pass and a fresh review weighs in, a person still does their own manual check and looks over the result before anything actually merges. Every task ends at a human gate. That gate has to clear before any work that depends on this task moves forward.

    The one thing to remember

    "Does it work?" and "is it built well?" are two separate questions, and a human is always the final gate. The agent proves function through a cheap-first validation pyramid; a fresh session judges quality; you decide what actually merges.
