Door one: starting from scratch

This is greenfield â€” an empty folder and an idea in your head. There's no existing code to learn from, so your first job is to get that idea onto paper: what are you building, for whom, and what does the first working version actually need to include? You have a conversation â€” with yourself, or with your agent â€” to pin down the goal, the phases of work, and how you'll know it succeeded, before a single file exists.

Once that's written down, it becomes the source of truth the rest of your project draws from. Every rule about how to build the thing, every convention you want followed, traces back to that early document. Nothing to reverse-engineer â€” you're defining the truth as you go.

Door two: stepping into someone else's code

This is brownfield â€” a codebase that already exists, whether it's three months old or thirty. Here the situation flips: there's no founding document to read, because the code itself is the documentation. Your first job isn't to invent intent, it's to uncover it. You (often with your agent's help) read through the structure, spot the patterns already in use, and figure out the conventions the project has settled into â€” how things are named, organized, tested, deployed.

For a large or sprawling codebase, this can mean sending an agent to explore several parts of the project at once and report back, rather than reading every file yourself line by line. Either way, the outcome is the same as door one: a clear picture of how this project works, which then becomes the foundation everything else builds on.

Door three: fixing one specific thing

Sometimes you don't need a founding document or a full tour of the codebase â€” you just need to fix a bug or close out one focused issue. This is the narrowest and fastest doorway. You don't skip understanding entirely â€” you still need enough context to know what's broken and why â€” but the scope is small and contained by definition: one problem, one fix, one check that it's actually resolved.

Because the scope is tight, this doorway often moves through the core loop the quickest of the three: a short research pass to find the actual cause, a light plan for the fix, the agent implementing it, and a validation step that proves the specific bug is gone.

The key move All three doors ask the same underlying question before any code is written: what's actually true here, and what am I trying to accomplish? Greenfield answers it by writing the truth down for the first time. Brownfield answers it by reading the truth out of existing code. A focused fix answers it in miniature, scoped to one problem. Why this convergence matters

This is genuinely good news for a beginner. It means you don't need to master three separate methodologies before you feel ready. Learn the core loop once, learn how to set up the surrounding system once, and you can point that same skill at a brand-new idea, an inherited codebase, or a single stubborn bug.
