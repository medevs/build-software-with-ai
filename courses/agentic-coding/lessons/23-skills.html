Building block #2, in depth

Skills (Reusable Workflows)

A skill is a workflow you write down once and trigger by name forever after. If rules are the block worth setting up first, skills are usually the block worth investing in most.

A skill is a named, repeatable workflow

Picture the workflows you already repeat by hand: planning a new feature, reviewing a diff before you commit, setting up a fresh environment, drafting a ticket the same way every time. Each of those is a multi-step process you currently re-explain — out loud, in your head, or in a fresh prompt — every single time you need it.

A skill packages that whole process into one document with a short name and description. Instead of re-explaining "first check the tests, then look at the diff, then flag anything risky, then write it up this way" for the tenth time, you just say the skill's name and the agent runs the entire routine, the same way, every time.

Why this is the highest-leverage block Almost any workflow you find yourself repeating can become a skill — planning, reviewing, onboarding, drafting, checking. One skill, written once, can save you from re-explaining the same steps for the rest of the project's life. Few investments pay back that fast.

How skills stay lean: progressive disclosure

Here's the part that makes skills practical instead of bloating your agent's context: the agent does not read a skill's full instructions until it actually decides to use it. Up front, all it sees is the skill's name and a one-line description — enough to recognize "this is relevant" without loading the whole thing into every conversation.

Only once the agent invokes a skill does it open the full document and follow the detailed steps inside. This two-tier pattern — a tiny always-visible summary, a larger on-demand body — is called progressive disclosure, and it's the same idea that keeps a rules file lean (see Lesson 22). You can have dozens of skills sitting around without any of them weighing down a task that doesn't need them.

Always visible

A short name plus a one-line description of what the skill does and when to reach for it — small enough to keep dozens of these around at no real cost.

Loaded on demand

The full step-by-step instructions, examples, and detail — only read once the agent (or you) actually triggers that specific skill.

Where a skill's value comes from

The payoff isn't just saving typing. A skill written once and used repeatedly means the workflow gets done the same way every time — same checks, same order, same output shape — instead of drifting a little each time you explain it slightly differently under time pressure. Consistency is what makes a workflow trustworthy enough to stop watching closely.

It also compounds. Skills are behavior, not codebase-specific truth, so a well-written skill for reviewing changes or drafting a plan tends to travel cleanly from one project to the next — unlike a rules file, which has to be rebuilt for each codebase.

The one thing to remember

Any workflow you find yourself repeating is a candidate to become a skill — write it once, and progressive disclosure keeps it from weighing down every other task. This is usually the highest-leverage building block you can invest in.
