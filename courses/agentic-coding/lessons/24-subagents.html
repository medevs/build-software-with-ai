Subagents

A subagent is a specialist you can hand a heavy job to â€” with its own separate context â€” so its digging doesn't clutter the conversation you're actually trying to get work done in.

A worker with its own workspace

Your main agent has one context window: everything you've told it, everything it's read, everything it's tried so far, all sitting in the same working memory. That's fine for a normal task. It gets crowded fast when the job is heavy â€” exploring a large unfamiliar area of the codebase, digging through many files looking for one answer, or critically reviewing a finished change.

A subagent solves this by being a separate specialist with its own context and its own tools. You delegate it one focused job. It goes off, does the heavy digging entirely in its own workspace, and hands back only a tight summary â€” the answer, not the messy trail of files and dead ends it took to get there. Your main conversation stays clean, exactly as it was before you delegated.

The core payoff A subagent's isolated context is what makes it useful. All the noise of a big search or a deep review happens somewhere else â€” you only ever see the conclusion.

Where subagents pay off first

Two situations come up constantly once you start looking for them:

Splitting up a big exploration

On a large, unfamiliar codebase, one agent trying to understand everything at once runs out of room. Fan the exploration out across several subagents, each covering a different slice, and have them report back short summaries you can act on.

A fresh set of eyes for review

The agent that wrote a change is a bad judge of that same change â€” it's primed to think its own work is fine. A subagent with no memory of writing the code has no such bias, so it makes a far more honest reviewer.

The classic example: a code-reviewer subagent

The clearest instance of this pattern is a subagent whose only job is to be critical. After a change is implemented, instead of asking the same agent that wrote the code "was that good?", you hand it to a separate reviewer subagent â€” one with a fresh context and instructions to look hard for problems.

This connects directly to the review step covered back in Lesson 13: the agent proving its own code runs is one kind of check. Having a separate, unbiased reviewer judge whether the code is well built is a different, harder-to-fake kind of check â€” and it only works because the reviewer isn't the same mind that wrote the code in the first place. You wouldn't want a student grading their own homework; the same logic applies here.

A simple way to decide Ask yourself: is this job big enough, or sensitive enough to bias, that I'd want it done somewhere separate from my main conversation? If yes, that's a job for a subagent.
