Choosing Your Coding Agent

You've spent this whole course building a context layer and a repeatable loop. Neither one cares which coding agent runs it. This lesson is about picking the tool without over-thinking it.

The tool is the base, not the system

Picture your setup as layers stacked on top of a model. At the very bottom sits a coding agent â€” a terminal or editor-based assistant that actually reads files, writes diffs, and runs commands on your behalf. On top of that you've built your context layer: the rules, skills, and connections that make the agent understand your project. On top of that runs the core loop you practiced in earlier modules.

The coding agent is the one layer in this whole stack you don't build. You choose it, the way you'd choose an editor or a terminal emulator. Popular options right now include Claude Code, Codex, Cursor, and GitHub Copilot, with more arriving every few months. They differ in interface, defaults, and which model they lean on â€” but every one of them is just a harness for the same underlying idea: an AI that can read your codebase, propose changes, and execute commands.

Why this matters Because the context layer and the core loop live above the tool, they transfer. Rules written for one agent translate to another with small naming changes. A quick, neutral survey of the landscape

You don't need to master every option â€” you need to recognize the shape of the category. A few well-known coding agents, treated evenly:

Claude Code

A terminal-based coding agent with a strong slash-command surface, built-in support for project rules files, sub-tasks, and connections to external tools. A common default for developers who like working from the command line.

Codex

OpenAI's coding agent, with its own conventions for rules files and sub-tasks. Functionally it covers the same ground as other agents in this category â€” proof that the underlying system isn't locked to any single vendor.

Cursor and others

Editor-native agents (Cursor), IDE extensions (GitHub Copilot), and newer entrants all compete on the same axis: how well they read context, how fast they iterate, and how much control they hand you over what runs.

Every one of these accepts some form of always-loaded project rules, some way to break work into smaller sub-tasks, and some way to connect to outside tools and data. That's not a coincidence â€” it's the shape the whole category has converged on, because it's the shape a reliable system needs.

Beginner criteria for picking one

If you're choosing for the first time, don't overthink it. A short, practical checklist:

  • Where do you already work? A terminal-first agent fits naturally if you live in a terminal; an editor-native one fits if you never leave your IDE.
  • Can it read and write a rules file? If it can't be given always-loaded project context, you'll be re-explaining yourself every session.
  • Does it support running commands and checking results? Your Validate step needs the agent to run tests and read the output, not just generate text.
  • What does it cost, and how does that fit your usage? Subscription limits and pricing models vary; pick something you can use daily without hesitation.
The real lesson Whichever one you pick, the context layer and the core loop you've already learned move with you. Switching agents later is a translation exercise, not a rebuild.
Don't over-invest here It's tempting to spend a week comparing agents before writing a line of code. Resist it. Pick a reasonable option today, build your system on top of it, and revisit the choice in a month once you actually know what you need from a tool.
