The app's memory

What Is a Database

Close an app, reopen it a week later, and your messages, your order history, your saved settings are all still there. That doesn't happen by accident — a database is where all of it lives.

A database remembers so the app doesn't have to

While an app is running, it keeps things in the computer's short-term memory — fast, but wiped clean the moment the app closes or the device restarts. A database is separate storage built to hold information permanently: it survives closing the app, restarting the phone, even the app being updated to a new version. When you open your banking app and see last month's transactions, the app isn't "remembering" them itself — it's asking a database to hand them over.

Almost every app you'd call "real" — one that has accounts, orders, messages, posts — has a database behind it. The app itself is mostly a set of instructions for showing and changing data. The database is the data.

Why this matters for you When you're judging whether an AI-built app is solid, one of the first questions is "where does this store its data, and does that survive a restart?" If the answer is "nowhere, really," the app will quietly lose things.

Tables: like a spreadsheet, one sheet per kind of thing

The most common way a database organizes information is in tables — and a table is very close to a spreadsheet sheet. Each table holds one kind of thing: a "Users" table, an "Orders" table, a "Products" table.

Rows = records

Each row is one specific item — one user, one order. If you have 4,000 customers, the Users table has 4,000 rows.

Columns = fields

Each column is one piece of information every row has — name, email, signup date. Every row in the table fills in the same columns.

That's a "Users" table with two rows. A real app usually has several tables that connect to each other — an "Orders" table that says which user placed which order. We'll build that connective tissue in the next lesson.

Why not just use a spreadsheet?

A spreadsheet can store rows and columns too, so it's a fair question. It falls apart the moment an app needs any of these:

Many people at once

A spreadsheet gets confused — or corrupted — when two people try to change it at the same second. A database is built for thousands of people reading and writing at the same time without stepping on each other.

Speed at scale

Searching a spreadsheet with ten million rows for "orders from last Tuesday" is slow. A database is built to answer that in a fraction of a second, even at huge size.

Rules it enforces

A database can refuse to save an order with no customer attached, or two users with the same email. A spreadsheet will happily save nonsense.

Reliability

If the power cuts mid-save, a database is designed to not leave data half-written. A spreadsheet offers no such promise.

A spreadsheet is a great tool for one person exploring numbers. A database is the tool for an app that many people depend on to be correct, every time.

Two families, one sentence each

You'll hear two broad families of database mentioned: "SQL" databases, which use strict, predefined tables like the ones above (great when your data has a clear, stable shape), and "NoSQL" databases, which store more flexible, loosely-shaped records (handy when the shape varies a lot). Neither is "better" — they're suited to different jobs, and we're simplifying heavily here just so the terms don't surprise you later.

Red flags to catch
  • The AI stores real user data only in memory or in a plain file with no database at all, "for now."
  • Nobody can tell you which table a piece of information — like a user's order history — actually lives in.
  • The app lets you save something the data clearly shouldn't allow, like an order with no customer.
Ask your AI
  • "Where is this app's data actually stored, and will it survive a restart?"
  • "What tables does this app use, and what does each one store?"
  • "What stops two people from saving conflicting changes to the same record at the same time?"

The one thing to remember

A database is the app's permanent memory, organized into tables — rows of records, columns of fields — built to stay correct with many people using it at once. Everything else the app does is really just reading from, or writing to, that memory.
