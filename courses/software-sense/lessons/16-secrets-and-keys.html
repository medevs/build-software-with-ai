Don't tape your house key to your front door

Secrets & Keys

Every real app holds a few things that must stay hidden â€” and the single most common way beginners lose control of their app is putting those things somewhere anyone can see.

Every app holds secrets

A secret is any piece of information that would let someone else do damage if they got hold of it. Most apps hold a handful without even thinking of them that way:

API keys

A kind of password that lets your app use a paid service â€” sending email, processing payments, calling an AI model. Whoever has the key can use that service as you, and you get the bill.

Database passwords & tokens

The password that lets your app read and write its own data, and small credential strings ("tokens") that prove your app is allowed to talk to another service. Anyone with these can reach your data directly.

The #1 beginner mistake

Here's how it usually goes wrong: someone is building quickly, so they paste a real API key straight into the code â€” right next to the line that uses it, because that's the easy place to put it. Then they push that code to a public repository (an online, shared copy of the project's code, visible to anyone if it's set to "public"). The key is now sitting in plain view on the internet.

This is not a rare, unlucky event. Bots constantly scan public code repositories specifically looking for patterns that look like API keys. It typically takes minutes, not days, for a leaked key to be found and used â€” often to run up enormous bills on someone else's paid service, quietly, in the background, before anyone notices.

Pasting a secret into your code and pushing it publicly is like taping your house key to your front door "just for now." It works, right up until someone walks by and notices.
Red flags to catch
  • A real API key, password, or token appears directly inside a code file.
  • The project's repository is public and nobody has checked it for secrets before pushing.
  • A leaked key was quietly replaced, but the old one was never actually disabled ("revoked").

The fix: keep secrets out of the code

The fix is simple in concept: secrets live outside the code, in separate configuration values called environment variables. Think of environment variables as sticky notes attached to the computer running the app, not to the recipe (the code) itself. The code just says "look up the sticky note named API_KEY and use whatever's on it" â€” it never contains the actual value. That means the code can be shared, viewed, or pushed publicly without ever exposing what's on the sticky notes.

  1. Store each secret as an environment variable, kept in a file or settings screen that's separate from the code itself.
  2. Make sure that file is explicitly excluded from anything pushed to a shared or public repository (most project setups have a place for this, often a file literally named for "things to ignore").
  3. If a secret is ever accidentally exposed, don't just delete it â€” revoke it (turn the old one off entirely) and issue a fresh one, since a deleted line of code doesn't undo who already saw it.
Why this matters for you AI tools will happily write a secret straight into your code if you ask them to "add the API key here" â€” they're following your instructions literally, exactly as Lesson 1 described. Keeping secrets out of code is a habit you have to hold, not something to assume the AI will catch on its own.
Ask your AI
  • "Is there any API key, password, or token written directly in this code? Where?"
  • "Set this up so the secret comes from an environment variable instead of being hardcoded."
  • "Is the file holding my secrets excluded from what gets pushed to the repository?"
