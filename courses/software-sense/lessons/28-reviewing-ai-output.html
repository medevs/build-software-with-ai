Reviewing AI's Work

This is the lesson everything else led to. You don't need to read code to review it well — you need to ask the right questions, in the right order. Here they are.

You review with questions, not syntax

You will probably never read a line of code the AI writes and understand it directly — and you don't need to. What you can do is ask the AI targeted questions about its own work and judge the answers, the same way you'd ask a contractor to walk you through renovation work you didn't do yourself. Each question below maps to a module you've already completed. You're not learning anything new here — you're assembling what you know into one pass.

The capstone review checklist

AskWhy it mattersFrom
Does it do what I asked — including the edge cases, not just the happy path? A demo proves the happy path once. Real users hit empty fields, duplicates, and invalid input constantly. L10
Is there any test covering this, and does it actually run? A test is a promise the behavior was checked at least once, automatically, not just eyeballed. L11
Is there anything that looks like a secret — a password, API key — sitting in the code itself? Secrets in code get copied, shared, and leaked. They belong somewhere the code never shows. L14
Does every sensitive action check who is asking before doing it? Missing permission checks are one of the most common ways real apps get broken into. L15
Does anything trust input from a user (or another system) without checking it first? Untrusted input is the raw material of almost every security incident. L14
What data does this store, and is that data modeled and backed up sensibly? Data that's shaped wrong or never backed up is expensive, sometimes irreversibly, to fix later. L7, L9
Will this hold up if a lot more people use it — or cost too much if it does? What works fine for you alone can slow to a crawl, or get expensive, under real traffic. L23, L26
Is this readable and consistent enough that changing it later won't be a nightmare? Software gets changed constantly. Messy code makes every future change slower and riskier. L29

How to actually run this, in practice

  • Paste the checklist to your AI, one section at a time, and ask it to answer honestly against the actual code — not a summary of what it meant to do.
  • Ask it to show you, not just tell you. "Show me the part of the code that checks for a duplicate email" is a better question than "did you handle duplicates?"
  • Treat a confident answer as a starting point, not a verdict. If something feels hand-wavy, ask it to point to the exact spot that proves the claim.
  • Keep a short list of what you couldn't verify — those are the open risks you're consciously accepting, not ones you missed.
    • Why this matters for you This checklist is the same one a technical reviewer runs in their head, just translated into plain questions. Running it doesn't require you to become technical — it requires you to not skip the questions because the demo looked good.
    Red flags to catch
    • The AI answers "yes, it's secure / tested / handled" with no specific example to back it up.
    • You only tried the happy path yourself before calling the review done.
    • Nobody has looked at what data gets stored, or whether it's backed up, even once.
    Ask your AI
    • "Walk me through what happens for each edge case in my definition of done — show me, don't just tell me."
    • "Is there anywhere untrusted input reaches this code without being checked first?"
    • "If a hundred times more people used this tomorrow, what's the first thing that would break or get expensive?"

    Every one of these questions is also collected as a ready-to-paste prompt in the resources page's "Ask your AI" prompt pack — worth bookmarking so you're never starting a review from a blank page.

    The one thing to remember

    You review AI's work with questions, not code-reading — one question per module you've learned, and evidence, not confidence, as the bar for "yes." This checklist is the whole course in one pass.
