"It works" isn't the finish line

Maintainability & Debt

Software isn't built once and left alone — it gets changed, over and over, for as long as it's used. Whether that's easy or a nightmare depends on a choice you can influence today.

Software is never really "finished"

Once software ships, the requests don't stop: fix this, add that, handle one more edge case, connect it to one more thing. Every piece of real software you've ever used has been changed dozens or hundreds of times since it first launched. That means the real question isn't just "does it work today" — it's "how hard will it be to change safely tomorrow?"

Two pieces of software can do the exact same thing today and be worlds apart on that second question. The difference is called maintainability: how easy the software is to understand, fix, and extend later, by anyone — including a future AI session that doesn't remember today's conversation.

Why this matters for you You may never open the code yourself. But every future change — a new feature, a bug fix, a request to an AI six months from now — either builds smoothly on what's there, or fights it. Maintainability is what decides which one happens.

Technical debt: the bill that comes due later

Technical debt is the future cost of a shortcut taken today. Like financial debt, a little of it, taken on purpose, can be a reasonable trade — ship the simple version now, clean it up next week. The trouble is debt taken on by accident, never tracked, and never paid down: every shortcut compounds, and eventually even small changes become slow and risky because nobody can safely touch the pile without something else breaking.

Debt isn't the shortcut itself — it's the shortcut nobody wrote down and nobody ever came back for.

What maintainable software actually looks like

Readable

Named clearly enough that "what does this do" has an obvious answer, even for something built months ago.

Consistent

Similar problems are solved the same way throughout, instead of a different approach every time a new piece was added.

Documented

The non-obvious decisions — why this, not the simpler option — are written down somewhere, not just remembered by whoever built it.

Not copy-pasted everywhere

The same logic lives in one place. Fixing a bug means fixing it once, not hunting down every copy.

None of these require you to judge code yourself. You can ask your AI directly whether each one is true, the same way you asked review questions in the previous lesson.

Why this matters more, not less, with AI

An AI coding agent can generate an enormous amount of code very fast — far faster than a person typing it by hand. That's the appeal. It's also the risk: a mountain of code produced in an afternoon can just as easily be a mountain of messy code, and messy code at ten times the normal speed is still a liability, just one that arrived faster. Volume was never the goal. Software that can keep being changed safely was always the goal — AI just makes it easier to lose sight of that while you're moving fast.

The fix isn't slowing down. It's asking for maintainability explicitly, the same way you'd name security or cost as a constraint in Lesson 27 — because left unstated, speed is what gets optimized for by default.

Red flags to catch
  • The same logic appears to be copy-pasted in several different places instead of written once.
  • Nobody, including the AI, can explain in plain English why a non-obvious piece was built the way it was.
  • Every new feature request seems to take longer than the last one, for no clear reason.
Ask your AI
  • "Is any of this logic duplicated elsewhere? If so, can it live in one place instead?"
  • "What's the one part of this you'd want a future developer — or a future you — to understand before changing it?"
  • "Did you take any shortcuts here I should know about, even ones that are reasonable for now?"

The one thing to remember

Software is changed constantly, and messy software gets exponentially harder and riskier to change — so ask for maintainable, not just working, and ask for it on purpose. "It ran once" was never the finish line; this lesson is the reminder that "it stays easy to change" is.
