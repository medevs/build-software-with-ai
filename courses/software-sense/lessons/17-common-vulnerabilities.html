The greatest hits, in plain English

Common Vulnerabilities

You don't need to become a security researcher. You just need to recognize the shape of the three mistakes that show up in breach after breach — because they all trace back to the same root cause.

The common thread: trusting input it shouldn't

Every vulnerability in this lesson comes from the same place: the app trusted something a user typed, uploaded, or sent, and treated it as safe instructions or safe data without checking it first. Lesson 14 called this "never trust user input" — here's what it looks like when that rule gets broken.

Injection: tricking the app into running your instructions

Injection happens when an attacker types something into an ordinary-looking box — a search bar, a login field, a web address — that isn't just data, but is crafted to be understood as a command by the system underneath. If the app doesn't check the input carefully, it ends up following the attacker's instructions instead of just treating the text as, say, someone's name.

A simple mental picture: imagine a form letter that says "Dear [Name], your balance is [Amount]." If "Name" is supposed to be filled in with plain text but someone instead writes a new instruction in that blank — and the system that assembles the letter can't tell the difference — the new instruction gets carried out. That's injection: input pretending to be data, but actually acting as a command.

Exposed data: reachable by changing a number

Exposed data is private information that turns out to be reachable simply by editing part of a web address or a request the app makes, without the app double-checking who's allowed to see it. This is the same disaster described in Lesson 15 — an order, profile, or file that should be private, but is actually shown to anyone who guesses or changes an ID number, because the app never rechecked ownership.

Cross-site scripting: content that attacks other users

Cross-site scripting (XSS) happens when an app lets someone submit content — a comment, a profile bio, a review — that isn't properly checked, and that content later gets shown to other users in a way that lets it act instead of just display. Imagine a comment section where, instead of leaving a normal comment, someone leaves something designed to run as an active instruction the moment another visitor loads the page — quietly acting on that visitor's behalf, in their browser, without them doing anything except viewing the page.

Injection

Input tricks the app's own systems into running the attacker's instructions.

Exposed data

Private info is reachable just by changing a number or address, with no ownership check.

Cross-site scripting rounds out the set: malicious content aimed not at the app itself, but at the other people who view it.

Where this comes from These three are part of the OWASP Top Ten, the industry's list of the most common, most damaging security mistakes in real software — a good name to recognize if you ever see it mentioned in a security tool or a developer's notes.
Red flags to catch
  • User-typed text gets combined directly into a command or lookup without being checked first.
  • Changing an ID number in a web address shows you information that isn't yours.
  • Content one user submits (a comment, a bio) is shown to other users without being checked or cleaned first.
Ask your AI
  • "Is any user input combined directly into a command or database lookup without being checked first?"
  • "If I change an ID number in this request, could I see or edit something that isn't mine?"
  • "Is user-submitted content (comments, bios) properly checked before it's shown to other users?"

The one thing to remember

Injection, exposed data, and cross-site scripting all come from the same root cause — the app trusted input it shouldn't have — so checking every input before it's used is the single habit that prevents most of this module's worst-case scenarios.

Reflect & discuss Next time you type something into a search box or comment field, notice: are you typing plain data, or could that box be tricked into treating your text as an instruction? Share what you notice in the course discussions.
