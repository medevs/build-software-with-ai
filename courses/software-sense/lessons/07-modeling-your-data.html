The decision everything else is built on

Modeling Your Data

Before an app can show anything on screen, someone has to decide what pieces of information it stores, and how those pieces connect to each other. That decision is called data modeling — and it's the most expensive thing in the whole app to get wrong.

Data modeling is deciding what you store and how it relates

Data modeling means answering two questions before a single table gets built: what pieces of information does this app need to remember, and how are those pieces connected? A simple online store needs to know about users, and about orders, and — crucially — that each order belongs to a user, and each order contains one or more items. That web of "this belongs to that" is the model.

It sounds obvious once it's said out loud. In practice, it's where most of the important thinking in building an app actually happens — including with AI. An AI can generate the tables and the code around them in seconds. It's much slower, and much more valuable, to get someone to think hard about whether "an order belongs to exactly one user" is even true for this business (what about gift orders? shared accounts?).

The foundation of a house

Think of the data model as the foundation and floor plan of a house, laid before the walls go up. You can repaint a wall in an afternoon. You can swap the flooring in a weekend. But moving a load-bearing wall, or discovering the plumbing was routed wrong, means tearing into everything built on top of it.

Screens and buttons are the paint. The data model is the foundation.

Every screen, every feature, every report the app will ever produce is built assuming the data is shaped a certain way. Change the shape later, and all of it may need to be touched — the tables, the code that reads them, and any data already saved in the old shape.

Why this matters for you When an AI offers to "just start building" a feature, it's worth pausing to ask what data model it's assuming. A few minutes spent questioning the shape now can save weeks of rework once real data is sitting inside it.

A good model versus a messy one

Say you're modeling a simple bookstore order. Here's the difference between a clean model and a messy one:

Good: separated and connected

A Users table, an Orders table linked to a user, and an Order Items table linked to an order — each item naming one book and a quantity. Adding a second book to an order just adds another row. Nothing else changes.

Messy: crammed into one place

One giant "Orders" table with columns like book_1, book_2, book_3 for up to three books per order. Someone orders a fourth book, and now there's nowhere for it to go without changing every row that came before.

The messy version isn't "wrong" in a demo — it can work for a first test. The problem shows up later, exactly when it's most expensive: real customers have already placed orders, and now the shape itself needs surgery while protecting data that's already there.

Red flags to catch
  • Nobody can describe, in one sentence, how two of the app's main things (like users and orders) relate.
  • The model has a fixed number of "slots" for something that's naturally unlimited (book_1, book_2, book_3…).
  • A small new feature request ("let a user save multiple addresses") suddenly requires reshaping existing tables.
Ask your AI
  • "Explain the data model for this app in plain English — what are the main things, and how do they connect?"
  • "What happens to this model if a user needs to have more than one of [X]?"
  • "Is any of this shape hard-coded in a way that will break if the business rules change?"

The one thing to remember

Data modeling is deciding what the app remembers and how those pieces connect — and because everything else is built on top of it, it's the one part worth slowing down for. Get the shape roughly right early, and the rest gets easier. Get it wrong, and every later change gets harder.

Reflect & discuss Think of an app with a "one slot" limit that annoyed you — one phone number, one address, one payment method — when you clearly needed more than one. That's usually a data model that didn't plan for it. Share your example in the course discussions.
