Define the finish line before you run the race

What "Done" Means

"Done" feels obvious until you try to say it out loud. Most software problems start here — nobody agreed what finished actually looks like.

"Done" is a definition, not a feeling

When you ask an AI to "build a signup form," it will build something. It will run. A box will appear, you'll type a name and an email, and it'll say "success." That can feel done. But done for whom, doing what, under what conditions?

A requirement is a plain statement of what the software must do. Not how — just what. "A visitor can create an account with an email and password" is a requirement. It sounds simple, but it's already doing real work: it tells you what to build and, just as importantly, what to check afterward.

Without a written requirement, "done" quietly becomes "whatever the AI happened to produce." That's not a definition — that's luck.

Why this matters for you You don't need a thick spec document. You need one or two sentences per feature, written down before you ask the AI to build it. That small habit is the difference between steering the software and being steered by whatever came out first.

The happy path is the easy 20%

The happy path is the scenario where everything goes exactly right: a real name, a valid email, a strong password, one click, done. Demos almost always show the happy path — it's fast, it looks good, and it's the first thing any builder (human or AI) gets working.

Real users don't stay on the happy path. They paste their email twice by accident. They leave a field blank. They type an address with no @ sign. They try to sign up with an account that already exists. None of that is unusual — it's just Tuesday for any software with real users.

Edge cases: the weird inputs that reveal reality

An edge case is an input at the boundary of what's expected — unusual, extreme, or invalid, but entirely realistic. A short list to picture, using that same signup form:

Edge caseWhat actually happens if you never asked
Empty — no email typedMight crash, might silently create a broken account
Huge — a 5,000-character nameMight break the page layout or overflow storage
Wrong — "not-an-email" as the emailMight get accepted and never receive a confirmation
Zero / negative — age field set to 0 or -5Might pass validation that only checks "is it a number"
Duplicate — signing up twice with the same emailMight create two accounts, or silently overwrite one
Malicious — someone pastes code into the name fieldMight get stored and later run as instructions — a real security risk

None of these are exotic hacker tricks. They're what happens the first ordinary week your software is live. A demo that never tried them hasn't been tested — it's only been shown once, in ideal conditions.

Writing a tiny definition of done

You don't need engineering training to write one. A useful definition of done for a feature has three short parts:

  • The requirement. One sentence: what must this do? "A visitor can create an account with an email and password."
  • The edge cases you care about. A short list: empty fields, duplicate email, obviously invalid email, a password that's too short.
  • What "handled" means. Not "crash" or "silently do the wrong thing" — instead, a clear message telling the person what went wrong.

    • Hand that whole thing to the AI along with your feature request. You're no longer asking it to guess what matters — you're asking it to build to a spec you defined, and you now know exactly what to test afterward.

    Red flags to catch
    • The feature was declared "done" the moment the happy path worked once.
    • Nobody can name what happens on an empty, duplicate, or malicious input — because nobody asked.
    • The requirement only exists in your head, not written anywhere the AI or a teammate can see it.
    Ask your AI
    • "Here's my requirement: [one sentence]. What edge cases should we handle before this is done?"
    • "What happens right now if this field is left empty, or the input is a duplicate?"
    • "Show me what the user sees when something goes wrong, not just when it goes right."

    Check yourself

    What is a "requirement," in plain terms?

    Why do edge cases matter if the demo already worked?

    What should a tiny "definition of done" include?

    The one thing to remember

    A demo proves the happy path works once. "Done" means you defined the requirement and the edge cases first, and checked both. Say what "done" means before you build — not after something breaks.
