Sending mail, and waiting for the reply

Client & Server

"It's slow." "It's down." Almost every complaint like that is really about one relationship: your device asking a distant computer for something, and waiting for the answer.

You ask, something else answers

Your phone or laptop, running the frontend from last lesson, is called the client. It's the one asking for things. Somewhere else — often in a data center you'll never see — sits the server: an always-on computer running the backend, waiting to answer.

The whole exchange works like ordering something by mail:

  • You write an order and send it off. In software, this is a request — the client asking the server for something ("show me my messages," "save this photo").
  • Your letter travels over a delivery network. In software, that's the internet — cables, towers, and routers carrying the request to the right server.
  • The server reads the request, does the work, and mails something back. That's the response — the answer the client receives and shows you.

    • Every single thing an app does — loading a page, posting a comment, checking your balance — is one or more of these request-and-response round trips happening, usually in well under a second.

    Why this matters for you Nothing your app does is instant or free. Every tap sends a letter out and waits for a reply. That round trip is where slowness, errors, and "why did this fail" almost always come from.

    The API: a menu of what you're allowed to ask for

    A client can't just ask a server for anything, in any form — there has to be an agreed set of requests the server understands and is willing to answer. That agreed set is called an API (application programming interface). Think of it as the menu a waiter hands you: you can't order anything you like, you order from the list of dishes the kitchen has agreed to make.

    The API is the waiter: it only accepts requests the kitchen has agreed to handle, phrased the way the kitchen expects.

    When an AI "builds an API" for your app, it's writing down that menu — which requests the server will accept ("get my order history," "add an item to my cart") and exactly what it needs from you to do it. Anything not on the menu, the server refuses.

    "It's slow" or "it's down" — where to look

    SymptomWhere to look first
    Whole app won't load, for everyoneThe server is probably down or overloaded
    Just your device is stuck spinningYour connection to the internet
    App loads but one action never finishesThat one request — likely a slow or broken part of the backend

    Because the client and server are separate machines, either one can be the problem — and so can the delivery network between them. That's also why an app can feel instant for you and unusably slow for someone with a weak connection, even though the server is doing exactly the same work both times.

    Red flags to catch
    • "It's not our servers' fault" said with no actual evidence of where the request failed.
    • A feature that works fine for the builder but is never tested on a slow or unreliable connection.
    • An API that will quietly accept requests it was never designed to handle safely.
    Ask your AI
    • "When I do this action, what request gets sent to the server, and what response comes back?"
    • "What does this API actually accept — and what happens if I send it something outside that?"
    • "If the server is slow or down, what does the user see right now?"

    The one thing to remember

    Your device is the client sending letters; a server somewhere else writes back — and "it's slow" or "it's down" almost always means a problem in that request, that response, or the delivery in between.
