Proof that keeps working, not just a good first impression

Testing, Explained

You checked it once and it worked. Six weeks from now, will it still? A test is how you find out without crossing your fingers.

A test is a small, automated check

A test is a short piece of software whose whole job is to check that another piece of software behaves the way it's supposed to. You (or the AI) write it once. From then on, it runs automatically — in seconds — and tells you pass or fail.

Think of it like a smoke detector. You don't check for smoke by wandering the house sniffing the air every hour. You install a detector once, and it keeps checking, forever, without you thinking about it. A test is the same idea, aimed at your software instead of your house.

Recall from the last lesson that "done" means a requirement plus its edge cases. A test is simply that definition of done, turned into something the computer can check for you, over and over, instead of you eyeballing it once.

Why this matters for you "It worked when I tried it" is a moment in time. A test is a standing promise: this still works, checked automatically, every time anything changes. Those are very different kinds of confidence.

Three sizes of test, in plain English

Tests come in a few common shapes, mostly differing in how much of the software they check at once:

Unit test

Checks one small piece in isolation — like "does this function correctly calculate a discount?" Fast, narrow, and the first line of defense.

Integration test

Checks that several pieces work correctly together — like "when the order is saved, does the confirmation email actually get sent?" Catches problems that show up only when parts connect.

There's a third, broader kind:

End-to-end test

Acts like a real user clicking through the actual product — open the site, add an item to the cart, check out, see the confirmation. Slowest to run, but closest to what a real person actually experiences.

UnitOne small pieceVery fastLogic mistakes in a single function

You don't need to memorize the taxonomy. What matters is the idea underneath all three: each one is a small, repeatable proof that something still works, simplified here to keep the picture clear — real testing setups have more nuance than three tidy boxes.

Why AI-written code needs tests more, not less

It's tempting to think: the AI wrote it, it looks clean, it must be fine. But you didn't write it — which means you can't fully vouch for what it actually does in every situation, only what you saw it do once. That gap is exactly what tests are for.

A test doesn't ask you to read and understand every line of code. It asks a much simpler question: given this input, did we get the expected output? You can define the expected output — the requirement — even without being able to write the code yourself. That's a review you're fully qualified to do.

Tests as a safety net for tomorrow's changes

The real payoff of tests shows up later, not on day one. Software keeps changing — a new feature, a fix, a request to the AI to "also add this." Every change risks quietly breaking something that used to work.

With tests in place, that risk becomes visible immediately: change something, run the tests, and anything that broke turns red right away — before a real user ever sees it. Without tests, the same broken thing might sit undiscovered for weeks, waiting for an actual person to stumble into it.

Red flags to catch
  • "It looked right when I tried it" is being treated as equivalent to "there's a test for it."
  • New features get added with no mention of tests for the edge cases from lesson 10.
  • Nobody re-runs the tests after a change — they were only ever run once, at the start.
Ask your AI
  • "Write tests for this feature that cover the edge cases we listed, not just the happy path."
  • "Run the existing tests and tell me if this change broke anything."
  • "In plain English, what does each of these tests actually check?"

The one thing to remember

A test turns "it worked once" into "it keeps working" — and because you didn't write the AI's code yourself, that proof matters more, not less. Tests are the safety net under every future change.
