What the app knows right now

State & Storage

"It forgot my cart" is one of the most common software complaints there is. Almost always, it traces back to the same mix-up: something the app should have written down permanently was only ever held in its head.

"State" is what the app currently knows

State is a word for everything an app currently knows about what's going on: are you logged in? What's in your cart? Which tab is open? Every one of those facts is a small piece of state, and the app has to keep it somewhere. The whole question this lesson answers is: somewhere, but where, exactly â€” because different somewheres behave very differently.

Four places state can live

The database

Permanent. Survives closing the app, restarting the device, months passing. This is where anything the user would be upset to lose belongs â€” orders, saved posts, account details.

A session

Lasts only until you log out or leave â€” a temporary "you are logged in as Priya" note the app keeps for as long as you're actively using it, then discards.

Files

Storage for things like uploaded photos or documents â€” not tucked inside a database table, but saved as their own files, with the database usually just keeping a note of where to find them.

Memory (in-process)

The most temporary of all: information held only while a specific screen or feature is actively running, gone the instant it closes â€” like text you've typed but not yet submitted.

Each one trades permanence for speed. Memory is instant but vanishes almost immediately. The database is slower to read from but practically permanent. Sessions and files sit in between.

Why mixing these up causes real bugs

"It forgot my cart" is usually one specific mistake: the cart was kept only in memory or a session, when the user expected database-level permanence. Add an item, close the app before checking out, come back tomorrow â€” and if the cart lived only in short-term state, it's gone, even though nothing "went wrong" from the computer's point of view. It did exactly what it was told; it just wasn't told to remember that far.

Too permanent

Saving something to the database that should have been temporary â€” like every keystroke of an unsent message â€” wastes space and can even leak information the user never meant to keep.

Too temporary

Keeping something only in a session or memory that the user clearly expects to survive â€” like a shopping cart, or a half-finished order â€” causes exactly the "it forgot" complaint.

Getting this right isn't a technical detail â€” it's a judgment call about user expectations, which is something you can absolutely weigh in on without writing a line of code.

A quick look ahead: caching

One more term you'll hear: caching, which means keeping a fast, temporary copy of information that's really stored somewhere else (usually the database), so the app doesn't have to fetch it the slow way every single time. It's a close cousin of the "where does state live" question â€” but it comes with its own tricky failure mode (showing you stale, out-of-date information), so it gets a full lesson of its own later in the course.

Why this matters for you When you hit a bug where the app "forgets" something, the first useful question isn't "what's broken" â€” it's "where was this supposed to be stored, and was it actually stored there?" That single question resolves a surprising share of real bugs.
Red flags to catch
  • Something the user would expect to survive closing the app â€” a cart, a draft, a setting â€” is only ever kept in memory or a session.
  • Sensitive information (like a full card number) is kept around in storage longer than it needs to be.
  • Nobody can tell you, for a given piece of data, which of the four places it actually lives.

  • "If the user closes the app right now, what will they lose?"
