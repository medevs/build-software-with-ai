The instructions did something other than intended

Bugs & Debugging

Software doesn't "glitch" or have a bad day. When it misbehaves, some instruction was wrong or missing — and finding which one is a skill anyone can learn.

A bug is a gap between "should" and "does"

A bug is any case where software does something other than what it was intended to do. Recall from lesson 1: a computer follows instructions exactly, with no common sense. So if it's misbehaving, the instructions it was given were wrong, incomplete, or didn't account for something — a case, an edge, an assumption nobody wrote down.

This reframes what a bug actually is. It's not the computer "acting up." It's a precise, findable gap between what the instructions say and what they should have said. That gap can always be located, because the computer is doing exactly what it was told — the trick is figuring out what it was actually told versus what you meant.

Why this matters for you You don't need to read code to reason about a bug. You need to reason about behavior: what should have happened, what actually happened, and where those two split apart. That's a plain-English skill, not a technical one.

Debugging is detective work, not guessing

Debugging is the process of finding and fixing a bug. Good debugging follows a repeatable sequence, the same one a detective uses on a case:

  • Reproduce it. Make the bug happen again, on purpose, following exact steps. If you can't trigger it reliably, you don't understand it yet.
  • Find the cause. Trace backward from where it went wrong to the instruction that caused it — the missing check, the wrong assumption, the edge case nobody handled.
  • Fix it. Correct the instruction — and check the fix doesn't quietly break something else that used to work.

    • Skipping step one is the most common mistake. It's tempting to guess at a fix for something you never actually reproduced. That's not debugging — that's hoping.

    "I can't reproduce it" is a real problem, not an excuse

    If a bug can't be reliably triggered, it can't be reliably confirmed as fixed either. "It seems fine now" after a guessed-at fix is not the same as "we found the actual cause and corrected it." The bug may simply be waiting for the same rare conditions to line up again.

    This is exactly why tests matter: a bug that gets a test written for it can never silently come back unnoticed. The test becomes a permanent, automatic check that this specific mistake stays fixed.

    A good bug report is a gift to your future self and the AI

    Whether you're describing a bug to a coworker or to an AI, the same short report does the job. It needs three things:

    PartWhat it answersExample
    Steps to reproduceExactly what you did, in order"Go to checkout, leave the coupon field empty, click Pay"
    ExpectedWhat should have happened"Payment should go through with no coupon applied"
    ActualWhat actually happened"Page shows an error and the Pay button stops working"

    Vague reports like "checkout is broken" force the AI (or a person) to guess at all three of these before any real debugging can start. A precise report skips straight to step two — finding the cause — because reproducing it is already done.

    Red flags to catch
    • A fix gets applied for a bug nobody could actually reproduce first.
    • Bug reports say only "it's broken" with no steps, expected result, or actual result.
    • A bug gets called "fixed" with no test added, so it could silently return unnoticed.
    Ask your AI
    • "Here are the exact steps to reproduce this, what I expected, and what actually happened — what's the likely cause?"
    • "Once you fix this, write a test so this specific bug can't come back unnoticed."
    • "What else in the code makes the same assumption that caused this bug?"

    The one thing to remember

    A bug is a findable gap between what the instructions say and what they should say — and reproducing it exactly is the first, non-skippable step to fixing it for good.
