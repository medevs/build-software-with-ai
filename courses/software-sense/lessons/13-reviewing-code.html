A second look, before the instructions go live

Reviewing Code

You don't need to write code to review it well. You need to ask the right questions and actually read the answers.

Code review is a second pair of eyes, on purpose

Code review is a deliberate second look at a set of instructions before they go live, asking three plain questions: is this correct, is this clear, and is this safe? It happens after something is built but before real users touch it â€” the last checkpoint where a problem is cheap to catch, instead of expensive to clean up afterward.

Here's the part that surprises people: reviewing doesn't require being able to write the code yourself. It requires being able to ask sharp questions and actually evaluate the answers â€” a skill that's entirely learnable without ever touching a programming language.

Why this matters for you When an AI writes the code, you become the reviewer by default â€” there's often no one else in the loop. That makes this the single most valuable skill in this course: not writing software, but knowing what to ask about it before you trust it.

Three questions that do most of the work

Is it correct?

Does it actually do what the requirement said, including the edge cases â€” empty, duplicate, huge, malicious â€” not just the happy path?

Is it clear?

Can the AI explain, in plain English, what this piece does and why â€” in a way that actually makes sense to you? If the explanation is foggy, that's a signal, not a formality.

Is it safe?

What happens with a wrong or hostile input? Does a mistake fail loudly and safely, or silently do the wrong thing? We go deeper on this in the next lesson.

Ask for the plain-English explanation â€” then actually read it

The single highest-leverage review habit: ask the AI to explain what it just built, in plain English, as if to someone who's never coded. Then genuinely read the answer, sentence by sentence, and ask yourself: does this match what I actually asked for?

This works because it forces the explanation to surface at the level you can judge â€” behavior and intent â€” rather than syntax you can't. If the explanation mentions handling something you never discussed, or skips something you clearly asked for, that mismatch is exactly what review is for catching.

Spaced recall: the demo trap, again

Back in lesson 1 you learned the most important idea in this course: a demo working once is not the same as being ready for real users. Review is where that idea becomes an action. It's the checkpoint that stops "it ran once, in a demo" from quietly becoming "we shipped it" without anyone asking the harder questions in between.

Red flags to catch
  • Nobody asked the AI to explain its own work in plain English before it shipped.
  • The explanation you got was vague, and you accepted it anyway instead of asking again.
  • The only thing reviewed was whether it ran â€” not whether it was correct, clear, and safe.
Ask your AI
  • "Explain what this does in plain English, as if I've never coded â€” what does each part handle?"
  • "Walk me through what happens on a wrong, empty, or malicious input here."
  • "What's the one part of this you're least confident about, and why?"
