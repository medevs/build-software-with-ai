Changing the foundation without bringing down the house

Migrations & Backups

Apps grow, and their data model grows with them. The question isn't whether the shape of your data will need to change — it's whether you change it safely, or you find out the hard way what you lost.

A migration is a careful, tracked change to the data's shape

Back in Lesson 7, we compared the data model to a house's foundation — cheap to plan carefully up front, expensive to redo once real data is sitting on top of it. A migration is exactly that redo, done as safely as possible: a written, step-by-step, tracked change to the shape of the database — adding a column, splitting a table, renaming a field — done in a way that doesn't damage the data already there.

The word "tracked" matters. A good migration is a recorded, repeatable set of steps — not someone quietly clicking around in the database by hand. That record means the same change can be tested first, applied the same way every time, and undone if something goes wrong.

Adding a room

Adding a new column, like letting users add a profile picture, is usually a low-risk migration — existing rows just get a blank or default value for the new field.

Moving a wall

Splitting one table into two, or changing what a column means, is a high-risk migration — every existing row has to be reshaped to fit, without losing or scrambling anything.

Backups: copies for when something goes wrong anyway

A backup is a regular, separate copy of the database, kept so that a mistake, a bad migration, a hardware failure, or an attack doesn't mean the data is gone forever. Think of it as the software equivalent of photographing your passport before a trip: you hope you never need it, and the one time you do, nothing else matters more.

Backups and migrations are a pair for a reason: a migration is the moment data is most at risk of being damaged, which is exactly why you'd want a recent backup sitting there first, just in case.

Why this matters for you "We'll add backups later" is one of the most common things said right before real data gets lost for good. Backups aren't a nice-to-have polish item — they're the safety net that has to exist before real users' data does.

The golden rules

  • Never lose user data. Any change to the data's shape should be checked, before it runs for real, for whether it could delete or corrupt information that's already there.
  • Test migrations before running them for real. Run the change against a copy of the data first, confirm it does what it's supposed to, then apply it to the real thing.
  • Keep backups you've actually tried restoring. A backup nobody has ever restored from is an assumption, not a safety net — the only way to know it works is to have practiced getting data back out of it.
    • Red flags to catch
    • A change to the database's shape gets run directly on real user data with no test first.
    • Backups exist "in theory," but no one has ever actually restored one to check it works.
    • "We'll add backups later" is said about an app that already has real users on it.
    Ask your AI
    • "Walk me through this migration step by step — could it delete or damage any existing data?"
    • "Has this migration been tested against a copy of real data before running it for real?"
    • "How often are backups taken, and when was one last actually restored to confirm it works?"

    The one thing to remember

    Migrations let you change your data's shape as the app grows, and backups protect you when something goes wrong anyway — treat both as required safety equipment, not optional polish. Test the change, and test that you can recover, before real users' data is on the line.
